# 260608_1%reserve.md — LSP 채널 유동성 튜닝 및 송금 보호 조치 보고서

이 보고서는 50M sat 채널 자동 생성의 비효율성을 개선하고, 채널 reserve 부족으로 인한 강제 폐쇄(Force-Close) 방지 조치 및 50만 sat 이상 결제 펜딩 장애를 해결한 모든 과정과 핵심 원리를 기술적으로 분석하여 기록한 문서입니다.

---

## 1. 50M sat 자동 생성 -> 적정 수준 (4x) 채널 스케일링 및 플러그인 버그 수정

### [배경 및 문제점]
기존 LSP 서버는 사용자가 소액을 입금받더라도 항상 ~50M sat에 이르는 과도한 규모의 채널을 자동 개설하여 LSP 측의 자금 효율성을 크게 저해하고 있었습니다. 또한, 수신자의 자체 자금(`peerFunding`) 기준으로 4배를 펀딩하도록 수정한 코드에서는 온더플라이(On-The-Fly) 채널 개설 시 수신자 자금이 `0 sat`이기 때문에, LSP의 추가 펀딩이 `0 sat`이 되어 채널이 무조건 최소 용량(100만 sat)으로 고정되는 계산 버그가 존재했습니다.

### [수정 사항 및 과정]
1. **LSP 플러그인 (`OpenChannelInterceptor.scala`) 수정**:
   * 기존: `val peerFunding = o.openChannelNonInitiator.open.toOption.map(_.fundingAmount)`를 기준으로 4배 계산.
   * 수정: 수신자 자체 자금이 아닌, 실제로 외부에서 들어오는 입금 요청액(`req.requestedAmount`)을 기준으로 4배의 유동성을 공급하도록 공식 변경.
   ```scala
   val requestedAmount = req.requestedAmount
   val lspFunding = requestedAmount * 4
   val finalFunding = if (lspFunding.toLong < 1000000) fr.acinq.bitcoin.scalacompat.Satoshi(1000000) else lspFunding
   ```
2. **반영 결과**:
   * 50만 sat 입금 시, 기존의 100만 sat 최소 용량 대신 `50만 * 4 = 200만 sat` LSP 펀딩을 포함하여 **약 250만 sat 규모의 넉넉한 채널**이 동적으로 개설됩니다.
   * 이를 통해 다음 결제 시 수신 한도(Inbound Liquidity) 부족에 따른 불필요한 Splice 동작을 예방할 수 있습니다.

## 2. 1% Reserve 부족으로 인한 채널 붕괴 방지 (지갑 검증 로직 추가)

### [배경 및 문제점]
라이트닝 네트워크 규격(BOLT #2) 상 각 채널은 총 용량의 최소 1%(기본값)를 예치금(Reserve)으로 남겨두어야 합니다. 사용자가 잔고를 전부 소모하여 예치금 1% 미만으로 송금할 경우, LSP 서버 측에서 프로토콜 위반으로 판단하여 채널을 즉각 **강제 폐쇄(Force-Close)**하고 온체인 트랜잭션을 날리는 치명적인 붕괴 현상이 발생했습니다.

### [수정 사항 및 과정]
클라이언트 지갑(Phoenix 앱) 단에서 송금 화면 진입 및 전송 버튼 클릭 시, 수수료를 포함한 전송 금액이 채널 예치금(1%) 요건을 침해하는지 사전 검증(Pre-flight Validation)하는 로직을 추가했습니다.

1. **대상 소스코드**:
   * [SendToBolt11.kt](file:///root/md/phoenix/phoenix-android/src/main/kotlin/fr/acinq/phoenix/android/payments/send/bolt11/SendToBolt11.kt) (일반 결제)
   * [SendOfferView.kt](file:///root/md/phoenix/phoenix-android/src/main/kotlin/fr/acinq/phoenix/android/payments/send/offer/SendOfferView.kt) (BOLT12 Offer 결제)
2. **핵심 검증 로직**:
   * 활성화된 commitments 채널의 총 펀딩 금액의 1%(`fundingAmount / 100`)를 합산하여 `totalReserveMsat`을 구한 후, 송금 시 남는 잔고가 이 예치금보다 작을 경우 에러 메시지를 띄우며 결제를 원천 차단합니다.
   ```kotlin
   val totalReserveSat = activeChannels.sumOf { it.commitments.latest.fundingAmount.toLong() / 100 }
   val totalReserveMsat = totalReserveSat * 1000L

   activeChannels.isNotEmpty() && (localBalanceMsat - currentAmount.toLong() < totalReserveMsat) -> {
       context.getString(R.string.send_error_reserve_insufficient) // "Channel reserve is insufficient for this payment."
   }
   ```

## 3. 50만 sat 이상 전송 불가 장애 해결 (`max-htlc-value-in-flight`)

### [배경 및 문제점]
40만 sat는 정상 전송되나 50만 sat 송금 시 펜딩에 머무르는 현상이 발생했습니다. 운영서버 로그 추적 결과, Eclair의 기본 설정 파일(`reference.conf`)에 정의된 **`max-htlc-value-in-flight-percent = 45`** (채널 용량의 최대 45%까지만 한 번에 전송 가능) 설정 때문이었습니다. 100만 sat 채널에서 50만 sat(50%)를 보내려다 45만 sat 한도에 막힌 것입니다.

### [수정 사항 및 과정]
1. **운영서버 (`10.8.0.6`) 설정 수정**:
   * `/root/.eclair/eclair.conf` 파일에 HTLC 인플라이트 한도를 채널 용량의 100%로 상향하는 설정을 주입했습니다.
   ```hocon
   eclair.channel.max-htlc-value-in-flight-msat = 100000000000
   eclair.channel.max-htlc-value-in-flight-percent = 100
   ```
2. **서버 재기동**:
   * PM2 환경에서 Eclair 노드를 재시작하여 성공적으로 한도 제한을 제거했습니다.
   `pm2 restart eclair`

## 4. Zero-Conf 채널 vs 중간 확장 (Splice) 컨펌 대기 메커니즘 분석

동일하게 온체인 L1 트랜잭션이 요구됨에도 불구하고 두 메커니즘 간 대기 시간의 차이가 발생하는 기술적 상세입니다.

### A. 최초 채널 개설 시: Zero-Conf (0-Conf) 작동 원리
* **기술 정의**: 온체인 채널 개설 트랜잭션이 비트코인 블록체인(L1)에서 아직 1컨펌도 되지 않은(Mempool에 대기 중인) 상태임에도, 즉시 채널을 활성화하여 결제를 처리하는 기술입니다.
* **리스크 판단**: 0-Conf 채널 개설 시, 유동성을 제공하는 주체는 LSP(서버)입니다. 지갑 앱(폰B)은 신뢰할 수 있는 운영 주체인 LSP를 신뢰하므로, LSP가 트랜잭션을 이중지불(Double-spend)하여 먹튀하지 않을 것이라 보증하고 결제를 즉각 Fulfill(완료) 처리합니다.
* **결과**: 사용자는 L1 컨펌을 기다릴 필요 없이 즉각 결제 완료 화면을 보게 됩니다.

### B. 채널 용량 초과 추가 송금 시: Splice 및 L1 컨펌 대기 지연
* **기술 정의**: 이미 존재하는 채널의 뼈대(온체인 UTXO)를 허물고 더 큰 뼈대(신규 UTXO)로 교체하면서 자금을 추가 투입(Splice-in)하는 과정입니다.
* **리스크 판단**: 이때는 이미 채널 내에 **사용자의 실제 자금(기존 예치금)**이 들어가 있습니다. 만약 Splice 트랜잭션이 컨펌되지 않은 0-Conf 상태에서 새로운 장부를 승인해 주었다가, 해당 Splice 트랜잭션이 L1에서 취소(Double-spend 등)되면 채널 내부 장부(Commitment)가 롤백되지 않아 **기존 자금까지 통째로 유실되는 대참사**가 일어날 수 있습니다.
* **안전 규격**: 따라서 Eclair와 Phoenix 표준 프로토콜은 Splice 트랜잭션이 **L1 블록에 안전하게 최소 1~2 컨펌 이상 안착할 때까지 결제 HTLC의 릴레이를 보류(Pending)**시킵니다.
* **해결 조치**: 사설 개발망 테스트 시 블록 마이닝이 자동으로 일어나지 않으면 영원히 펜딩되므로, 비트코인 코어 노드에 `generatetoaddress` RPC 명령을 날려 강제로 블록을 생성(컨펌)해 주어야 펜딩이 해소됩니다.
