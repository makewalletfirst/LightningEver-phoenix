## 1. 4개 패치 (반드시 필요)

다시 구현할 때 반드시 이 4개 패치를 적용해야 합니다. 하나라도 빠지면 위 4가지 기능 중 하나가 실패합니다.

### 1-1. lightning-kmp: Peer.kt feerate fallback

**파일**: `modules/core/src/commonMain/kotlin/fr/acinq/lightning/io/Peer.kt` (`updateFeerates()` 함수)

**문제**: BitEver의 electrs는 블록이 적어서 `blockchain.estimatefee`가 항상 에러를 반환 → `onChainFeeratesFlow.value`가 `null` 유지 → ForceClose 흐름의 `currentOnChainFeerates()`가 **영원히 suspend**.

**증상**: ForceClose 버튼을 누르면 `(force) closing channel=…` + `error on command ForceClose: forced local commit` + `retrieving feerates`까지만 로그에 찍히고 commit tx가 브로드캐스트되지 않음. 채널 상태는 `Normal` 유지.

**패치**:

```kotlin
suspend fun updateFeerates() {
    client.getFeerates()?.let { feerates ->
        logger.info { "on-chain fees: $feerates" }
        onChainFeeratesFlow.value = OnChainFeerates(feerates)
    } ?: run {
        // DEV-BYPASS for BitEver: electrum cannot estimate fees on test chain.
        if (onChainFeeratesFlow.value == null) {
            val minFee = FeeratePerKw.MinimumFeeratePerKw
            logger.error { "cannot retrieve feerates! using minimum=$minFee as fallback" }
            onChainFeeratesFlow.value = OnChainFeerates(minFee, minFee, minFee, minFee)
        } else {
            logger.error { "cannot retrieve feerates! keeping previous=${onChainFeeratesFlow.value}" }
        }
    }
}
```

### 1-2. lightning-kmp: WaitForInit.kt Negotiating 복원

**파일**: `modules/core/src/commonMain/kotlin/fr/acinq/lightning/channel/states/WaitForInit.kt`

**문제**: 앱 재시작 시 `Negotiating` 상태가 복원되어도 그 안의 `publishedClosingTxs`가 다시 publish/watch되지 않음 → mutual close가 "syncing(negotiating)"에서 영원히 멈춤.

**패치 핵심**:

```kotlin
is Negotiating -> {
    // DEV-BYPASS: Re-publish and re-watch closing transactions on restart.
    val actions = buildList {
        addAll(unconfirmedFundingTxs.map { ChannelAction.Blockchain.PublishTx(it, ChannelAction.Blockchain.PublishTx.Type.FundingTx) })
        addAll(fundingTxWatches.map { ChannelAction.Blockchain.SendWatch(it) })
        cmd.state.publishedClosingTxs.forEach { closingTx ->
            add(ChannelAction.Blockchain.PublishTx(closingTx))
            add(ChannelAction.Blockchain.SendWatch(
                WatchConfirmed(cmd.state.channelId, closingTx.tx,
                    staticParams.nodeParams.minDepthBlocks,
                    WatchConfirmed.ClosingTxConfirmed)
            ))
        }
    }
    Pair(Offline(cmd.state), actions)
}
```

### 1-3. lightning-kmp: Commitments.kt + Transactions.kt musig2 nonce 정렬

**파일들**:
- `modules/core/src/commonMain/kotlin/fr/acinq/lightning/channel/Commitments.kt` (`fullySignedCommitTx`)
- `modules/core/src/commonMain/kotlin/fr/acinq/lightning/transactions/Transactions.kt` (`aggregateSigs`)

**문제**: simple_taproot_channels의 musig2 partial signature 합산 시 `publicNonces[i]`가 `sortedKeys[i]`와 인덱스가 일치해야 함. 기본 코드는 `[localNonce, remoteNonce]`로 고정되어 있어서, `Scripts.sort(...)` 결과로 remote가 first면 깨짐 → commit tx 서명 검증 실패.

**패치**: `Scripts.sort()`로 funding pubkey 정렬 후 정렬 순서에 맞춰 nonce도 함께 정렬.

```kotlin
val sortedFundingKeys = Scripts.sort(listOf(fundingKey.publicKey(), remoteFundingPubkey))
val orderedNonces = if (sortedFundingKeys.first() == fundingKey.publicKey())
    listOf(localNonce.publicNonce, remoteSig.nonce)
else
    listOf(remoteSig.nonce, localNonce.publicNonce)
val localSig = unsignedCommitTx.partialSign(fundingKey, remoteFundingPubkey, mapOf(), localNonce, orderedNonces).right!!
val signedTx = unsignedCommitTx.aggregateSigs(fundingKey.publicKey(), remoteFundingPubkey, localSig, remoteSig, mapOf()).right!!
```

`aggregateSigs`도 똑같이 partialSigs와 nonces를 sortedKeys 순서에 맞춰 정렬.

### 1-4. Eclair plugin: channel-funding-plugin

**파일**: `/root/eclair-plugins/channel-funding/`

**문제**: 외부 L1으로 swap-in이 들어와도 채널이 만들어지지 않고 즉시 `aborted`로 가는 현상.

**해결**: Eclair를 시작할 때 반드시 plugin JAR을 인자로 함께 로드해야 함.

```bash
./bin/eclair-node.sh /root/.eclair/plugins/channel-funding-plugin-0.13.1.jar
```

Plugin의 핵심 역할: swap-in 입금 감지 시 `AddFunding(50000000 sat)`을 응답하여 Eclair가 LSP 측 funding을 50,000,000 sat으로 보충 → 폰 입장에서는 자기 swap-in 2,500,000 sat과 합쳐서 ~52,500,000 sat 채널이 열림.

플러그인 없이 Eclair만 실행하면 `AddFunding(1 sat)`만 응답되어 dust 한계에 막혀 abort.

## 2. 자동 채널 생성 (외부 L1 swap-in)

### 2-1. 시퀀스

1. 폰에서 swap-in 주소(`bc1p…`) 표시
2. 외부 비트에버 지갑에서 이 주소로 송금 (예: 2,500,000 sat)
3. 폰의 SwapInWallet이 utxo를 감지 → `deeplyConfirmed` (3 conf) 도달
4. 폰의 Peer가 `requestChannelAdd` 시작 → `OpenDualFundedChannel` 송신
   - `fundingAmount=2499850 sat` (폰의 swap-in 입금 - mining fee)
   - `RequestFunding(requestedAmount=1 sat, …, paymentDetails=FromChannelBalance)` TLV 포함
5. Eclair plugin이 `RequestFunding` 보고 → `AddFunding(50000000)` 응답 → 합쳐서 ~52,500,000 sat 채널
6. interactive-tx 협상 → `TxComplete` 교환
7. musig2 nonce 교환 (위 1-3 패치 적용 상태여야 정상)
8. `CommitSig` 교환 → 채널 0-conf 활성화
9. funding tx 브로드캐스트 → confirm → 채널 `Normal`

### 2-2. 확인

```bash
# Eclair LSP에서
eclair-cli -p '비밀번호' -a 152.67.210.39:8080 channels
```

채널 상태가 `NORMAL`이면 성공.

## 3. Bolt12 송금 (폰 A → 폰 B)

### 3-1. 사전 조건

- 폰 A에 채널이 있어야 함 (송신자가 잔고 보유)
- 폰 B에 채널이 있어야 함 (수신자가 incoming capacity 보유)
- 두 폰 모두 LSP에 연결되어 있어야 함 (`PeerConnected`)
- 두 폰의 chainHash가 동일해야 함 (BitEver hash)

### 3-2. 흐름

1. 폰 B에서 receive 화면 → "Use a reusable Bolt 12 offer" → `lno1…` 표시
2. 폰 A에서 send 화면 → `lno1…` 입력 → 금액 입력 → "Send"
3. 폰 A가 LSP에게 onion message 송신 (offer의 introduction node = LSP)
4. LSP가 blinded path로 폰 B에게 onion message 라우팅
5. 폰 B가 invoice를 응답 (LSP를 경유)
6. 폰 A가 trampoline payment 송신:
   - First-hop fee: `feeBase=4 sat, feeProportional=4000` (`TrampolineFees`)
   - 1-hop trampoline (LSP가 직접 폰 B에게 전달)
7. 폰 B가 preimage로 정산 → 양쪽 채널이 commit number 증가

### 3-3. 함정 회피 (이번에 직접 겪은 것들)

| 증상 | 진짜 원인 | 대처 |
| --- | --- | --- |
| "fee is insufficient" UI 에러 | 실제로는 `route not found` (수신자 채널 없음) | 폰 B가 먼저 swap-in으로 채널 열어야 함 |
| 무한 "Waiting for channels..." | 폰 B 채널이 `Offline`/`Syncing` 상태 | LSP 재시작 후 30초 대기 → 둘 다 `Normal` 회복 후 재시도 |
| 송금이 안 되고 timeout | 폰 A 잔고 부족 (trampoline fee 포함) | 폰 A 잔고 ≥ 송금액 + 약 5 sat 여유 |

자세한 분석은 `260517_LN_2.md` 참고.

## 4. Mutual Close (지정 L1 주소)

### 4-1. 흐름

1. 폰에서 Settings → Channels → "Mutual close" → 회수 주소 입력 + feerate 선택
2. Phoenix가 `MutualCloseAllChannels(address, feerate)` intent 송신
3. `AppCloseChannelsConfigurationController`가 `peer.send(WrappedChannelCommand(channelId, ChannelCommand.Close.MutualClose(…)))`
4. KMP 채널 actor가 `Shutdown` 메시지 송신 → 상대편이 응답 → `ClosingComplete`/`ClosingSig` 교환
5. closing tx 브로드캐스트 → confirm → 지정 주소로 잔고 회수

### 4-2. 함정

- 패치 1-2(WaitForInit Negotiating 복원)가 없으면 앱 재시작 시 멈춤
- feerate가 너무 낮으면 (`min_feerate_sat_byte` 미만) LSP가 거부할 수 있음
- 지정 주소가 segwit이 아니면 ANYSEGWIT 옵션 위반으로 거부됨

## 5. Force Close

### 5-1. 흐름

1. 폰에서 Settings → Channels → "Force close" → 확인
2. Phoenix가 `ForceCloseAllChannels` intent 송신
3. KMP 채널 actor:
   - `(force) closing channel=…` 로그
   - `handleLocalError(cmd, ForcedLocalCommit(channelId))` → `error on command ForceClose` 로그
   - **`spendLocalCurrent()` → `currentOnChainFeerates()` 호출**
   - **(여기서 패치 1-1이 없으면 영원히 suspend!)**
4. `fullySignedCommitTx()` (패치 1-3 적용) → commit tx 빌드
5. `claimCommitTxOutputs()` → `LocalCommitPublished` 빌드 (claim-main-delayed tx 포함)
6. PublishTx 액션으로 commit tx 브로드캐스트
7. **Commit tx confirm 시 Phoenix UI는 "Complete" 표시**
8. **CSV 잠금 해제 후** (다음 단계 참고) claim-main-delayed tx 브로드캐스트 → final wallet으로 잔고 회수

### 5-2. CSV 잠금 (반드시 이해해야 할 점)

Force close 시 잔고가 즉시 final wallet에 들어오지 **않습니다**. Commit tx의 `to_local` output에는 `to_self_delay` 만큼의 CSV lock이 걸려있고, 이 블록 수가 지나야 KMP가 claim-main-delayed tx를 브로드캐스트할 수 있습니다.

현재 설정:

| 측 | to_self_delay (BitEver 기준) |
| --- | --- |
| 폰 → LSP에게 부과 (Open) | 2016 블록 |
| LSP → 폰에게 부과 (Accept) | 720 블록 |

폰이 force-close하면 자기 commit tx의 to_local에는 **LSP가 부과한 720 블록** CSV가 걸립니다. 그래서 약 720 블록 채굴 후 자동 회수.

### 5-3. 테스트 체인용 단축 (선택)

테스트 편의를 위해 to_self_delay를 줄이려면:

- **KMP 측** (`NodeParams.kt:259`): `toRemoteDelayBlocks = CltvExpiryDelta(2016)` → `144` 등으로 변경 (폰이 LSP에게 부과하는 값)
- **Eclair plugin 측**: Accept 시 `toSelfDelay`를 작게 (폰에게 부과하는 값) — 이걸 줄여야 폰 force-close 시 빨리 회수

둘 다 줄이면 다음 채널부터 효과. 이미 열린 채널엔 영향 없음.

## 8. 자주 발생하는 사고와 즉시 복구

### 8-2. "Bolt12 송금 시 fee is insufficient"

- 원인: 수신자 채널 없음 (UI 오타)

### 8-4. "force close 버튼 눌러도 반응 없음"

- 원인: 패치 1-1 빠짐 (feerate suspend)
- 해결: 위와 동일

### 8-5. "force close는 Complete인데 잔고가 안 들어옴"

- 원인: 정상! to_self_delay (720 블록) CSV 잠금
- 해결: 블록 채굴 대기, 또는 to_self_delay 줄이는 패치(섹션 5-3)

### 8-6. "musig2 signature verification failed"

- 원인: 패치 1-3 빠짐 (nonce 순서 깨짐)
- 해결: Commitments.kt + Transactions.kt 두 곳 모두 패치
